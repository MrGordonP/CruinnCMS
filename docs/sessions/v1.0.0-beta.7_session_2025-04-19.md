## Architecture Decisions

1. **Payments as separate module** — Forms don't handle payments directly. When `require_payment` is enabled, form submission redirects to `/payments/initiate?source=form_submission&source_id={id}`. This allows the payments module to serve other modules (events, membership) later.

2. **Payment options per form** — Each form can define multiple price tiers via `form_payment_options` (label, amount, currency). Submissions record which option was selected.

3. **Payment verification is manual** — payment_status tracks `not_required`, `pending`, `verified`, `rejected`. Verification by membership secretary, not automated gateway (Stripe webhook support is stubbed but not wired).

4. **CSRF meta tag in admin layout** — Added `<meta name="csrf-token">` so all admin JS can access the token for AJAX without needing a hidden form field.
