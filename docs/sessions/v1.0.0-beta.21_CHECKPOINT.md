# v1.0.0-beta.21 Session Checkpoint

## Purpose

Implement membership plan hierarchy support (groups and tiers) and add optional Subject linkage on plan records so plan definitions can be scoped to Subjects at source.

---

## Scope

### Included

- Membership plan hierarchy support:
  - `parent_plan_id` on `membership_plans`
  - grouped ordering and grouped rendering for plans/tiers
- Membership plan capability wiring in admin CRUD:
  - `is_group`
  - `max_members`
  - `parent_plan_id`
- Optional Subject linkage on plan definitions:
  - `subject_id` on `membership_plans`
  - active Subject selection in plan form
  - Subject display in plans list
- Validation updates for plan integrity:
  - group cannot be assigned as tier
  - no self-parenting
  - parent must exist and be group-enabled
  - group plans require `max_members >= 2`
  - selected Subject must exist

### Explicitly Not Included

- No new dependencies
- No instance-specific behavior
- No routing framework changes
- No membership subscription flow redesign

## Migration Notes

### 033: membership tiers

Adds `membership_plans.parent_plan_id` with:

- index: `idx_membership_plans_parent`
- FK: `fk_membership_plans_parent` -> `membership_plans(id)` (`ON DELETE SET NULL`)

### 034: membership plan subject

Adds `membership_plans.subject_id` with:

- index: `idx_membership_plans_subject`
- FK: `fk_membership_plans_subject` -> `subjects(id)` (`ON DELETE SET NULL`)

Both migrations are idempotent and follow the existing prepared-statement guard pattern.

---

## Behavioral Outcome

- Plans can now act as top-level groups and hold tier children.
- Tiers are visibly grouped under their parent plan in plans listing.
- Plan definitions can optionally carry Subject linkage (`subject_id`).
- Plan create/edit UI supports Subject assignment without making it mandatory.
- Existing plans remain valid with `NULL` parent/subject fields.

---

## Validation Snapshot

- Editor diagnostics report no syntax errors in touched PHP/SQL files.
- Membership plan form and plans list templates compile with new fields.
- Service-layer plan retrieval now returns Subject title (`subject_title`) via join.

## Notes for Testing

Suggested smoke checks:

1. Run migrations 033 and 034 in order.
2. Create a group plan with max members >= 2.
3. Create a tier plan linked to that group via Parent Group.
4. Assign a Subject to group and/or tier plan and verify persistence.
5. Open plans list and confirm grouped display and Subject column values.
6. Confirm validation errors for invalid parent/group combinations.
7. Confirm existing plans without parent/subject still render and can be edited.
