## Changes Made

### 1. Platform Install Form — Password Validation

**File:** `CruinnCMS/templates/platform/install.php`

**Problem:** Password mismatch only caught server-side; form values lost on validation error, forcing user to re-enter everything.

**Solution:**
- Added client-side password confirmation validation using HTML5 `setCustomValidity()`
- Live visual feedback: red hint appears when passwords don't match
- Form preservation: all field values (except passwords) retained via `<?= e($username ?? 'platform') ?>` pattern on validation failure
- JavaScript validation prevents form submission if passwords don't match

**Lines Modified:**
- 72-74: Password/confirm inputs with IDs for validation
- 75: `.password-match-hint` div for error display
- 79-97: Form inputs wrapped with session-based value preservation
- 119-138: Client-side validation logic (`checkMatch()`, event listeners)

---

### 2. Platform Controller — Form Preservation Logic

**File:** `CruinnCMS/src/Platform/Controllers/PlatformController.php`

**Changes:**
- `showInstall()` (lines 83-95): Now passes `$formData` array to template with preserved values from session
- `install()` (lines 96-128): Stores form data in `$_SESSION['install_form_data']` on validation error
- Form values cleared from session on successful install

**Result:** User doesn't lose database credentials, site name, or username when server-side validation fails.

### 3. Instance Discovery UI — State-Aware Actions

**File:** `CruinnCMS/templates/platform/dashboard.php`

**Problem:** Orphaned instance directories (e.g., `iga/` with deleted database) showed confusing "Bring Online" button despite having no working DB connection.

**Solution:** Split action buttons into three states based on `db_connected` flag:

#### Online Instance
```
[Open Admin] [ACP Settings] [DB Browser] [Take Offline]
```

#### Offline with Valid DB
```
[Bring Online] [DB Browser]
```

#### Orphaned (No DB Connection)
```
[Provision Database]
Database not found or connection failed
```

**Data Source:** `gatherInstanceData()` already returns `db_connected` bool (line 1656) — now properly utilized in template.

**Link Behavior:** "Provision Database" routes to `/cms/instances/new?restore={folder_name}` for potential pre-fill support (not yet implemented).

## Technical Notes

### Instance Discovery Mechanism

Instances are discovered via **filesystem scan**, not database registry:

```php
// PlatformController.php line 1565
$dirs = glob($instancesBase . '*', GLOB_ONLYDIR);
```

**Rationale:** Allows users to drop in instance backup directories and provision databases afterward. The platform doesn't maintain a canonical instance registry — it scans `instance/` on every dashboard load.

**Instance States:**
- **Orphaned:** Directory exists, DB connection fails (`db_connected=false`)
- **Offline:** Directory + working DB, no `.active` file (`db_connected=true`, `online=false`)
- **Online:** Directory + working DB + `.active` file present (`db_connected=true`, `online=true`)

### Database Creation Privilege

Platform install attempts `CREATE DATABASE IF NOT EXISTS`:
- **VPS/local:** Works automatically if root/privileged user
- **Shared hosting:** Requires manual database creation via cPanel/Plesk (platform falls back gracefully)
