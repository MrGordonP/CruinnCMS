## 5. Mailbox Module (new)

Full IMAP webmail module: `modules/mailbox/`

**Architecture:**
- IMAP server is permanent archive; DB stores header index only (bodies fetched live)
- Thread graph built from `In-Reply-To` / `References` headers — no subject matching
- Per-user read receipts with three-state: unread / partial (some holders read) / all read
- Passwords AES-256-CBC encrypted at rest using instance secret
- Access derived from `organisation_officers.user_id` — single login, role-derived mailbox access
- Admin users see all `imap_enabled` mailboxes

**Files created:**
- `module.php` — manifest with routes, migrations, acp_sections

`subscribers` ↔ `users` ↔ `members` — three independent entities. `members.user_id` is an optional bridge. Member rows are never created automatically by user registration or OAuth login. Membership is an explicit act (joining, or admin creation).

## CruinnCMS API — Canonical Conventions (confirmed this session)

These are the correct patterns. Any module or template that deviates will cause errors.

### Template escaping
```php
e($value)          // CORRECT — global helper
$this->escape($value) // WRONG — method does not exist
```

### Flash messages
```php
// Do nothing in individual templates.
// Flash is handled globally in templates/layout.php via $flashes.
// Auth::getFlash() does not exist. Auth::getFlashes() exists but layout calls it.
```

### Breadcrumb array format
```php
['Section', '/admin/section']           // CORRECT — positional
['label' => 'Section', 'url' => '...']  // WRONG — named keys
```
