### 8. Article Editor Controller Rewritten Cleanly — `CruinnCMS/modules/blog/src/Controllers/ArticleEditorController.php`
- Replaced the corrupted controller file with a clean plain-ASCII version.

### 9. Shared Editor Media Upload Fixed — `public_html/js/admin/utils.js`, `public_html/js/admin/media-browser.js`
- Root cause of the 403 upload failure: the shared editor exposes its CSRF token via `#editor-wrap[data-csrf]`, while the media browser helper was only checking hidden form inputs.
- `Cruinn.getCSRFToken()` now falls back to `#editor-wrap[data-csrf]` and then `meta[name="csrf-token"]`.
- Media browser uploads now send the CSRF token in the `X-CSRF-TOKEN` header as well as the multipart form body, matching the existing shared upload helper pattern.

## Resolved Bugs

### 1. Persistent 500 on `POST /admin/article-editor/{id}/action`

**Status:** Resolved.

**Actual root cause:** `CruinnCMS/modules/blog/src/Controllers/ArticleEditorController.php` on the live host was corrupted. Replacing the file cleanly fixed the editor save path.

**Related schema issue also fixed:** the blog module's old `schema.sql` created `article_edit_state` and `article_draft_blocks` with an obsolete structure, so migration `004_fix_editor_state_schema.sql` was added to rebuild them with the schema expected by the controller.

### 2. Media Upload 403 from Blog Post Editor

**Status:** Resolved.

**Actual root cause:** the shared editor page stores its CSRF token on `#editor-wrap[data-csrf]`, but the admin media browser helper was only looking for hidden form inputs. Uploads from the article editor therefore submitted an empty token and were rejected by the global CSRF middleware.

**Fix:** update `Cruinn.getCSRFToken()` to read the shared editor token source and make media-browser uploads send the CSRF header as well.
