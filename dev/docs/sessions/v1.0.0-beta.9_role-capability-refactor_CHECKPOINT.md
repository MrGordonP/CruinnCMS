# Session Checkpoint — v1.0.0-beta.9 Role Capability Refactor

**Date:** 2026-05-17  
**Focus:** Role-based authorization refactor (Stages 1–3) + production hotfix

## What Changed

This session completed the first three stages of the role capability refactor plan documented in `dev/docs/ROLE_CAPABILITY_REFACTOR.md`, moving CruinnCMS from hardcoded role slug checks to numeric role levels with granular admin area grants and widget dashboard canvases.

### Stage 1 — Auth API Cleanup (`cc6c030`)

**Modified:** `src/Auth.php`

**Removed deprecated methods:**
- `Auth::role()` — replaced with `Auth::roleLevel()` (returns highest numeric level)
- `Auth::hasRole($slug)` — replaced with level comparisons or `Auth::hasAnyRole()` for multi-role checks
- `Auth::requireRole($slug, $redirect)` — replaced with `Auth::requireLevel($minLevel, $redirect)`

**Added new methods:**
- `Auth::roleLevel()` — returns numeric level (0-100) of user's highest-level role
- `Auth::roleLevels()` — returns all role levels for multi-role users
- `Auth::hasAnyRole(array $roleSlugs)` — checks if user has any of specified roles
- `Auth::requireLevel(int $minLevel, ?string $redirect = null)` — enforces minimum role level
- `Auth::isAdmin()` — convenience wrapper for roleLevel() >= 90

**Refactored existing methods:**
- `adminMiddleware()` now uses `roleLevel() >= 90` instead of `hasRole('admin')`
- `roles()` simplified query to fetch role info without redundant slug-based checks

**Result:** Cleaner API surface, numeric-first authorization model, backward compatibility broken (fixed later).

### Stage 2 — Admin Area Grants (`fdaab4f`)

**New file:** `config/admin_areas.php`  
Registry of grantable admin sections. Each area has:
- `name` — display label
- `description` — explanatory text for grant UI
- `icon` — Font Awesome icon class
- `routes` — array of route patterns (`{module}/subpath/*`)
- `module` — (optional) module dependency; area hidden if module not installed

**Defined areas:**
- Blog, Events, Documents (`*` wildcard grants all document type folders)
- Media (`admin/media/*`)
- Menus (`admin/menus/*`)
- Forum, Mailbox, Mailout, Membership
- Organisation (high-level access to org module; distinct from position-based member access)
- GDPR, Payments, Social

**Schema changes:**

`schema/instance_core.sql` — Added table:
```sql
CREATE TABLE admin_area_grants (
  id INT AUTO_INCREMENT PRIMARY KEY,
  role_slug VARCHAR(50) NOT NULL,
  area_slug VARCHAR(100) NOT NULL,
  granted_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY unique_role_area (role_slug, area_slug),
  FOREIGN KEY (role_slug) REFERENCES roles(slug) ON DELETE CASCADE
);
```

`migrations/core/015_admin_area_grants.sql` — Instance migration for existing deployments.

**Modified:** `src/Auth.php`

**Added methods:**
- `Auth::getAreaSlugForUri(string $uri)` — Resolves current URI to area slug using config routes
- `Auth::hasAreaGrant(string $areaSlug)` — Checks if user's roles grant access to area
- `Auth::requireAdminArea(string $areaSlug, ?string $redirect = null)` — Middleware for area-specific access control

**Updated:** `Auth::adminMiddleware()`  
Now performs two-stage check:
1. User must be admin-level (roleLevel >= 90)
2. If an admin area applies to the current route, user must have a grant for that area

This allows granular delegation: a user with `admin` role can be restricted to only `blog` and `events` sections by granting only those areas.

**New controller methods:** `RoleAdminController`
- `areaConfig(string $roleSlug)` — GET `/admin/roles/{slug}/areas` — shows area grant checkboxes
- `saveAreaConfig(string $roleSlug)` — POST `/admin/roles/{slug}/areas` — saves area grants

**New template:** `templates/admin/roles/areas.php`  
Checkbox list of all available areas with current grants pre-checked. Info alert explains that `admin`-level roles see all areas regardless of grants (grants only apply to delegation scenarios).

**Result:** Fine-grained admin access control ready for delegation workflows (e.g., "Blog Editor" role with only blog/media access).

### Production Hotfix — Backward Compatibility Shims (`53e9134`)

**Problem:** After deploying Stage 1, live site threw 500 error:  
`Call to undefined method Cruinn\Auth::role()` in `src/BlockTypes/nav-menu/definition.php`

**Root cause:** Removed `Auth::role()`, `hasRole()`, `requireRole()` are still used in 100+ locations across modules (forum, drivespace, membership, organisation).

**Solution:** Added deprecated compatibility shims to `src/Auth.php`:

```php
/**
 * @deprecated Use Auth::roleLevel() instead
 */
public static function role(): ?string
{
    $roles = self::roles();
    return !empty($roles) ? $roles[0]['slug'] : null;
}

/**
 * @deprecated Use Auth::hasAnyRole() or direct role level comparisons
 */
public static function hasRole(string $roleSlug): bool
{
    return self::hasAnyRole([$roleSlug]);
}

/**
 * @deprecated Use Auth::requireLevel() instead
 */
public static function requireRole(string $requiredRole, ?string $redirect = null): void
{
    if (!self::hasRole($requiredRole)) {
        if ($redirect) {
            header("Location: $redirect");
            exit;
        }
        http_response_code(403);
        die('Access denied');
    }
}
```

**Also fixed:**
- `src/BlockTypes/nav-menu/definition.php` — Changed to use `Auth::roleLevel()` and numeric `min_role` comparison (future-proof)
- `src/Controllers/PageController.php` — Changed `Auth::hasRole('admin')` to `Auth::isAdmin()` (home page redirect logic)

**Plan:** Deprecated methods marked with `@deprecated` tags. Future work: migrate all module call sites to new API, then remove shims in beta.10 or beta.11.

**Result:** Live site stable, gradual migration path established.

## Lessons Learned

1. **Breaking changes need compatibility layers** — Should have added deprecated shims in Stage 1 instead of discovering the need via production 500 error
2. **Test on production-like environment first** — Local dev with no modules active didn't reveal the breakage
3. **Gradual migration is safer** — Keeping deprecated methods with clear `@deprecated` tags allows incremental module updates
4. **Numeric role levels are cleaner** — Logic like `roleLevel() >= 90` is more readable than `hasRole('admin')` and supports custom role hierarchies better
5. **Admin area grants are powerful** — Enables fine-grained delegation (e.g., "Events Coordinator" role with only events/media access)
6. **Widget dashboards need better naming** — "Canvas" vs "Dashboard" terminology could be clearer in UI (canvas = template, dashboard = rendered page?)
