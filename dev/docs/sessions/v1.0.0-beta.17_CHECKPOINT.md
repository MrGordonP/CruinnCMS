## Context

Following on from beta.16 which established `ModuleRegistry::registerCore()` and the core module pattern, this session completed the work of expressing all discrete core capabilities as first-class module registrations — with widgets, content providers, and dashboard cards fully wired into the editor's dynamic-include selectors.

---

## What Changed

### 1. Core Module Registrations Completed (`src/App.php`)

All six remaining core modules are now fully declared via `ModuleRegistry::registerCore()`. They are always-active, not tracked in `module_config`, and appear in `widgetCatalog()` and `contentProviderCatalog()` on equal footing with drop-in modules.

#### `accounts`
- **ACP section:** `/admin/users`
- **`template_path`:** `CruinnCMS/templates/`
- **`content_providers`:**
  - `accounts:account-information` — Account Information (created_at, last_login) → `public/account/information.php`
  - `accounts:account-details` — Account Details Form (display_name, email) → `public/account/details-form.php`
  - `accounts:account-password` — Account Password Form → `public/account/password-form.php`
- **`widget_providers`:**
  - `accounts:recent-activity` — Recent Activity feed → `admin/widgets/recent-activity.php`
  - `accounts:notifications-summary` — Notifications Summary (per logged-in user) → `admin/widgets/notifications-summary.php`
  - `accounts:member-profile` — Member Profile card → `admin/widgets/member-profile.php`
- **`widgets` callable:** `accounts_quick_link`, `accounts_status_summary` (total/active/last-30d user counts)

#### `pages`
- **ACP sections:** `/admin/pages`, `/admin/templates`
- **`template_path`:** `CruinnCMS/templates/`
- **`widget_providers`** (all wired to `DashboardService` static methods):
  - `pages:stats-overview` — Platform stats overview
  - `pages:communications` — Recent articles + draft count
  - `pages:comms-social` — Combined comms + social links
  - `pages:social-links` — Social media quick links
  - `pages:upcoming-events` — Upcoming published events
  - `pages:forum-recent` — Recent forum threads
  - `pages:active-discussions` — Active council discussions
  - `pages:council-stats` — Council document/discussion counts
  - `pages:recent-documents` — Recent council documents
- **`widgets` callable:** `pages_quick_link`, `pages_status_summary` (total/published/draft)

#### `menus`
- **ACP section:** `/admin/menus`
- **`widgets` callable:** `menus_quick_link`, `menus_status_summary` (menu count + item count)

#### `media`
- **ACP section:** `/admin/media`
- **`widgets` callable:** `media_quick_link`

#### `roles`
- **ACP section:** `/admin/roles`
- **`widgets` callable:** `roles_quick_link`, `roles_status_summary` (role count)

#### `groups`
- **ACP section:** `/admin/groups`
- **`widgets` callable:** `groups_quick_link`, `groups_status_summary` (group count)

### 2. `AuthController` Content Provider Methods (`src/Controllers/AuthController.php`)

Three new static methods added at the bottom of `AuthController`, implementing the module content provider contract:

- `contentProviderAccountInformation(array $settings, array $context): array`
  Queries `created_at, last_login` for the current logged-in user. Returns `[]` if not logged in.

- `contentProviderAccountDetails(array $settings, array $context): array`
  Queries `display_name, email` for the current logged-in user. Returns `[]` if not logged in.

- `contentProviderAccountPassword(array $settings, array $context): array`
  Returns `[]` (password form template needs no server-provided user data). Returns `[]` if not logged in.

All three produce a silent empty render when unauthenticated — no error output.

---

## Architecture note

`DashboardService` static methods (`statsOverviewData`, `recentActivityData`, etc.) are now claimed by core module `widget_providers` declarations. They are no longer a homeless internal API — they have a module owner and are addressable by key from the editor. The `admin/widgets/*.php` templates are reachable via the `pages` and `accounts` module `template_path`.

`widgetCatalog()` and `contentProviderCatalog()` in `ModuleRegistry` are agnostic to registration origin — core and drop-in modules appear identically in all editor selectors.
