## What Changed

### 1. Per-file "Pull from Repo" (source editor properties panel)

**Goal:** Allow in-browser file updates from GitHub without SSH.

- Removed inline pull buttons from the file tree (were misplaced/broken).
- Added pull capability to the **right-hand properties panel** instead.
- Panel shows: path, file type, size, last modified, writable status.
- "↓ Pull from Repo" button POSTs to `/cms/source/pull` → overwrites local file with `raw.githubusercontent.com` content.
- Protected paths (`config/`, `instance/`, `public/uploads/`, `public/storage/`) are blocked with an explanatory message rather than a button.

**Files:**
- `CruinnCMS/config/routes.php` — added `POST /cms/source/pull`
- `CruinnCMS/src/Platform/Controllers/PlatformController.php` — `platformSourcePull()` method
- `CruinnCMS/templates/platform/source-editor.php` — properties panel rework, file stats, pull form

---

### 2. Folder "Pull Folder from Repo" (recursive)

**Goal:** Pull an entire directory subtree in one click.

- Clicking a `<details>` folder node in the tree populates `#props-dir` in the right panel with the folder path.
- "↓ Pull Folder from Repo" triggers an AJAX `POST /cms/source/pull-dir` call.
- Server-side: recursively enumerates the repo directory via the GitHub Contents API, fetches each file via `raw.githubusercontent.com`, and writes it locally only if the file already exists (no phantom file creation).
- Results rendered inline: `✓ ok`, `— skipped` (not local), `✗ failed`.
- **URL encoding fix:** GitHub Contents API requires per-path-segment encoding (slashes must not be encoded). Fixed by splitting on `/`, encoding each segment, then re-joining before building the API URL.

**Files:**
- `CruinnCMS/config/routes.php` — added `POST /cms/source/pull-dir`
- `CruinnCMS/src/Platform/Controllers/PlatformController.php` — `platformSourcePullDir()` method
- `CruinnCMS/templates/platform/source-editor.php` — folder click handler, AJAX pull, results display

### 3. `platform_github_path_maps` — path translation DB table

**Goal:** The GitHub repo layout (`src/`, `templates/`, etc.) maps directly to the engine root. On cPanel, `public/` is deployed as `public_html/` at a different tree level. A path map table in the platform DB translates local relative paths to repo paths so pull works correctly regardless of deployment layout.

**Table:** `platform_github_path_maps` (`local_prefix`, `repo_prefix`, `sort_order`)  
**Longest-match wins** — `sort_order DESC` in the query.

**Default rows (corrected):**

| local_prefix    | repo_prefix     | sort_order |
|-----------------|-----------------|------------|
| `src/`          | `src/`          | 80         |
| `templates/`    | `templates/`    | 70         |
| `schema/`       | `schema/`       | 60         |
| `migrations/`   | `migrations/`   | 50         |
| `modules/`      | `modules/`      | 40         |
| `config/`       | `config/`       | 30         |
| `public/`       | `public_html/`  | 20         |

**Bug fixed this session:** Initial seeded values had a spurious `CruinnCMS/` prefix in all `repo_prefix` values (e.g. `CruinnCMS/src/` instead of `src/`). The GitHub repo root IS the engine root — there is no `CruinnCMS/` subdirectory in the repo. This caused HTTP 404 on both single-file and folder pulls.

**Fix:**
- `CruinnCMS/schema/platform.sql` — corrected INSERT values
- `CruinnCMS/schema/migrate_platform_github_paths.sql` — corrected INSERT values
- Live DB: `UPDATE platform_github_path_maps SET repo_prefix = REPLACE(repo_prefix, 'CruinnCMS/', '');`

**Files:**
- `CruinnCMS/schema/platform.sql` — `platform_github_path_maps` table + corrected seed data
- `CruinnCMS/schema/migrate_platform_github_paths.sql` *(new)* — migration for existing installs (creates both tables, seeds maps, self-registers)

### 4. `/cms/migrations` — platform migrations runner

**Goal:** Apply `schema/migrate_platform_*.sql` files to the platform DB from the browser.

- Lists all matching migration files with applied/pending status.
- "Apply All Pending" button applies everything not yet recorded in `platform_migrations`.
- Per-row "Apply" button for individual migrations.
- `platformMigrationState()` bootstraps the `platform_migrations` table if missing (safe for first-ever run).
- Glob pattern `migrate_platform_*.sql` intentionally excludes `migrate_page_tables.sql` (instance migration) and other non-platform files.
- DELIMITER-aware SQL executor (`platformExecSql()`) handles multi-statement migration files.

**Files:**
- `CruinnCMS/config/routes.php` — `GET /cms/migrations`, `POST /cms/migrations`
- `CruinnCMS/src/Platform/Controllers/PlatformController.php` — `platformMigrations()`, `platformRunMigrations()`, `platformMigrationState()`, `platformMigrationRows()`, `platformExecSql()`
- `CruinnCMS/templates/platform/migrations.php` *(new)* — migrations list UI
- `CruinnCMS/templates/platform/layout.php` — "Migrations" nav link added

```sql
UPDATE platform_github_path_maps SET repo_prefix = REPLACE(repo_prefix, 'CruinnCMS/', '');
```

Run in `/cms/database/query` against the **platform DB**.

## Lessons Learned

- The GitHub raw URL base is `https://raw.githubusercontent.com/MrGordonP/CruinnCMS/main/` — the repo root corresponds directly to the engine root. No subdirectory prefix.
- GitHub Contents API requires per-segment URL encoding — `rawurlencode($fullPath)` encodes `/` to `%2F` and breaks the URL. Must split, encode, rejoin.
- When a path-map table seeds with wrong data, the symptom is HTTP 404 from GitHub with the wrong path visible in the error message — which is exactly how we caught it.
