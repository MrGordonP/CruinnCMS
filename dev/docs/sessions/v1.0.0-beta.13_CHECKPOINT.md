### System Pages Block Rendering (COMPLETE)

**`src/Controllers/BaseController.php`**
- Added `renderSystemPage(string $slug, array $data)`. Pushes `$data` into `Template::globals()` via `addGlobal`, queries `pages_index` for the slug, calls `CruinnRenderService::setContext($data)` + `buildWithTemplate(...)`. Falls back to `$this->render('public/' . $slug, $data)` if no `pages_index` row found.

**`src/Controllers/AuthController.php`**

### OAuthService — Changed (Status: Unconfirmed, possibly unnecessary)

**`src/Services/OAuthService.php`**
- `enabledProviders()` changed from:
  ```php
  $cfg = App::config("oauth.{$provider}", []);
  if (!empty($cfg['client_id'])) { ... }
  ```
  to:
  ```php
  $settings = new \Cruinn\Services\SettingsService();
  $clientId = $settings->get("oauth.{$provider}.client_id", '');
  if (!empty($clientId)) { ... }
  ```
- **This change may be unnecessary.** `App::loadSettingsOverrides()` already reads ALL non-empty rows from `settings` at startup and merges them into `App::$config` via dot-notation. `App::config('oauth.google', [])` returns `['client_id' => 'xxx', ...]` correctly IF `loadSettingsOverrides()` ran successfully. Both approaches should yield the same result when the DB has the OAuth credentials.
- **Why it was made:** Attempted to fix OAuth buttons not appearing on live login page. Did not resolve the issue.

## What Is Still Broken

**OAuth buttons on live login page do not appear.**

The issue was never resolved in this session. All code paths were traced and confirmed correct:

- `oauth_icon()` is defined in the `namespace {}` (global namespace) block in `Template.php` — fully accessible from any included file.
- `enabledProviders()` reads from DB correctly via both `SettingsService` and `App::config()` (which gets DB data via `loadSettingsOverrides()`).
- `renderSystemPage` pushes `oauth_providers` into both `Template::globals()` AND `CruinnRenderService::setContext()`.
- The php-include closure merges both sources into `$vars` before `extract()`.
- `login.php` template is unchanged; the `$oauth_providers` check and `oauth_icon()` call are correct.

## Dead-Ends Chased (Wasted Session Time)

1. **Namespace of `oauth_icon()`** — Spent significant time suspecting it was in the `Cruinn` namespace and inaccessible from login.php's no-namespace context. It is in `namespace {}` (global). Non-issue.

2. **`EXTR_SKIP` blocking `$oauth_providers`** — Suspected EXTR_SKIP was preventing extraction because a same-named variable existed in closure scope. No same-named variable exists in scope before extract. Non-issue.

3. **php-include closure not receiving `$context`** — Suspected the closure parameter `$context` was missing. The parameter was already added; the real issue with 500 errors was the missing `array $context = []` signature on definition.php closures (which WAS fixed, but was a separate issue from OAuth).

4. **`loadSettingsOverrides()` not covering OAuth keys** — Suspected `FILE_ONLY_KEYS` was blocking OAuth keys from being loaded into `App::$config`. Confirmed: `FILE_ONLY_KEYS` only protects `paypal.client_secret` and `stripe.secret_key`. OAuth keys load fine. Non-issue.

5. **SettingsService re-read** — Changed `enabledProviders()` to use `SettingsService` thinking the original `App::config()` wasn't reading DB. It was. Both approaches are equivalent. Change was unnecessary and didn't fix anything.
