## What Changed

Completed Stage 4 of the role capability refactor: integrated the organisation_officers table to provide position-based authorization, allowing users who hold officer positions to inherit admin area grants and widget dashboard assignments from those positions.

### Core Enhancement — Position Authorization

**Problem:** The authorization system had role-based grants (Stage 2) and widget dashboards (Stage 3), but no way to grant access based on organisational positions. A user holding "Secretary" or "Treasurer" positions needed admin-level access to specific areas without being a full admin.

**Solution:** Implement `Auth::positionIds()` to query active organisation_officers rows for the current user, cache in session, and check position grants alongside role grants in `Auth::requireAdminArea()`.

### Modified: `src/Auth.php`

**Added methods:**

```php
public static function positionIds(): array
```
- Checks if organisation module is active
- Returns cached session value if present (`$_SESSION['user_position_ids']`)
- Loads from DB via `loadPositionIds()` if not cached
- Returns empty array if not logged in or module inactive

```php
private static function loadPositionIds(int $userId): array
```
- Queries `organisation_officers WHERE user_id = ? AND active = 1`
- Returns array of officer IDs ordered by sort_order
- Gracefully handles missing table (module not installed)
- Called at login and when `refreshPositionIds()` is triggered

```php
public static function refreshPositionIds(): void
```
- Public method to reload position IDs in session
- Call after modifying organisation_officers assignments
- Ensures immediate propagation of position changes

**Login integration:**
- Modified `Auth::attempt()` to set `$_SESSION['user_position_ids'] = self::loadPositionIds($user['id'])`
- Modified `Auth::loginById()` to set `$_SESSION['user_position_ids'] = self::loadPositionIds($user['id'])`
- Position IDs cached alongside role_level and group_level for performance

**Access control integration:**
- `Auth::requireAdminArea($slug)` already iterates position IDs and checks grants (implemented in Stage 2)
- No changes needed — position checking was future-proofed in Stage 2 scaffold

### Organisation Module Changes

**New Routes:**
```php
GET  /admin/organisation/officers/{id}/areas       → officerAreas()
POST /admin/organisation/officers/{id}/areas       → saveOfficerAreas()
GET  /admin/organisation/officers/{id}/dashboard   → officerDashboard()
POST /admin/organisation/officers/{id}/dashboard   → saveOfficerDashboard()
```

**New Controller Methods:** `OrganisationAdminController`

**`officerAreas(int $id)` — GET officer area grants UI**
- Loads officer record with user display name
- Fetches current grants from admin_area_grants (context_type='position', context_id=officer_id)
- Renders checkbox grid of all grantable areas from config/admin_areas.php
- Filters out areas requiring inactive modules

**`saveOfficerAreas(int $id)` — POST save area grants**
- Clears existing grants for this position
- Inserts new grants from `$_POST['areas']` array
- Calls `Auth::refreshPositionIds()` if current user holds this position
- Redirects back to officers list with success message

**`officerDashboard(int $id)` — GET dashboard assignment UI**
- Loads officer record
- Fetches assigned dashboard from context_dashboards (context_type='position', context_id=officer_id)
- Renders dropdown of available widget dashboard canvases
- Shows "Edit Blocks" link if dashboard assigned

**`saveOfficerDashboard(int $id)` — POST save dashboard**
- Removes existing dashboard assignment for this position
- Inserts new assignment if dashboard_id selected
- Sets created_by to current user
- Redirects to officers list with success message

## How It Works

### Position Assignment Flow

1. Admin navigates to `/admin/organisation/officers`
2. Creates or edits officer position, assigns a user account
3. Clicks "Areas" → selects "Blog" + "Events" → saves
4. Clicks "Dashboard" → selects "Council Dashboard" → saves

Grants stored:
```sql
INSERT INTO admin_area_grants (area_slug, context_type, context_id) VALUES
  ('blog',   'position', 23),  -- officer_id = 23
  ('events', 'position', 23);

INSERT INTO context_dashboards (context_type, context_id, page_id) VALUES
  ('position', 23, 45);  -- page_id = 45 is the Council Dashboard canvas
```

### User Login Flow

1. User logs in
2. `Auth::attempt()` → `loadPositionIds($userId)` → queries organisation_officers
3. Result: `[23, 47]` (user holds two officer positions)
4. Cached in session: `$_SESSION['user_position_ids'] = [23, 47]`

### Access Check Flow

1. User visits `/admin/blog`
2. `Auth::requireAdminArea('blog')` called
3. Checks:
   - `Auth::isAdmin()` → false (user is not admin-level)
   - Role grants → none
   - Position grants → iterates `[23, 47]`
     - Position 23 has grant for 'blog' → **access granted**

### Dashboard Resolution Flow

1. User visits `/admin/dashboard`
2. `DashboardService::resolveDashboardForUser($userId)` called
3. Resolution order:
   - User-specific dashboard → none
   - Position dashboards → checks position 23 → finds page_id 45
4. Renders Council Dashboard (page_id 45) via `renderWidgetCanvas(45, userContext)`
5. Widget blocks receive `_userContext['position_ids'] = [23, 47]` for personalized queries

## Known Limitations

**Session refresh:** Position grant changes require re-login to take effect unless `Auth::refreshPositionIds()` is called explicitly. Current implementation only refreshes if the current admin user holds the modified position.

**Proposed improvement:** Call `refreshPositionIds()` for affected user when saving grants, regardless of who is logged in. Requires looking up affected user's session or notifying on next login.

**Dashboard priority:** When user holds multiple positions, resolution uses `$positionIds[0]` (first position). No priority/sort_order logic yet.

**Proposed improvement:** Add `dashboard_priority` column to organisation_officers or use sort_order to determine which position's dashboard takes precedence.
